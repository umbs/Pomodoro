Pomodoro Architecture

Technical design document for the Pomodoro Timer app.

Overview

The app follows a simple three-layer architecture. The UI layer reads state reactively via Consumer<TimerProvider>. The state layer owns all logic and delegates to two services: audio playback and persistent storage.

Phase Transition Logic

if (_phase == TimerPhase.work) {
  _completedSessions++;
  if (_completedSessions % sessionsBeforeLongBreak == 0) {
    _phase = TimerPhase.longBreak;
  } else {
    _phase = TimerPhase.shortBreak;
  }
} else {
  _phase = TimerPhase.work;
}

Classes and Responsibilities

TimerPhase (enum)

lib/models/timer_state.dart

Three possible states: work, shortBreak, longBreak. Determines which duration to count down, which color to display, and which sound to play.

TimerSettings (immutable data class)

lib/models/timer_state.dart

Holds the four user-adjustable values. Uses copyWith() for immutable updates.

Persistence

shared_preferences (BSD license) — a key-value store. No database, no migrations, no schema.

What's Not Implemented (by design)

  • No database or migrations
  • No user accounts or cloud sync
  • No notification center / system notifications
  • No task naming or history log
  • No theming or dark/light mode toggle
  • No internationalization
  • No analytics or telemetry
